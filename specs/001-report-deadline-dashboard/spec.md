# Feature Specification: 报表报送监控看板（Report Submission Deadline Dashboard）

**Feature Branch**: `001-report-deadline-dashboard`  
**Created**: 2026-04-16  
**Status**: Draft  
**Input**: User description: "新加一个监控报表报送的看板，对一周内即将到期的报表报送进行提醒，防止报表报送逾期"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 查看报表报送截止日期看板 (Priority: P1)

作为 Maker 或 Checker 用户，我希望登录系统后能看到一个独立的"报送监控"看板页面，以卡片/列表形式展示所有已配置截止日期的报表及其当前报送状态，方便我一目了然地了解哪些报表即将到期、哪些已经逾期。

**Why this priority**: 这是整个功能的核心价值——让用户在一个集中视图中看到所有报表的报送截止情况，没有看板就无法实现任何提醒。

**Independent Test**: 可以通过登录任一角色并导航至看板页面，确认看到报表列表及其截止日期和状态标签来独立测试。

**Acceptance Scenarios**:

1. **Given** 系统中存在已配置截止日期的报表，**When** 已登录用户导航至"报送监控"看板页面，**Then** 页面以卡片或列表形式展示每个报表的名称、描述、截止日期、距截止日剩余天数、以及当前报送状态（未报送/已报送/已审批/已逾期）。
2. **Given** 系统中没有任何配置了截止日期的报表，**When** 用户打开看板页面，**Then** 页面显示空状态提示"暂无待报送的报表"。
3. **Given** 看板已加载，**When** 用户查看某个报表卡片，**Then** 卡片显示该报表最近一次 ReportRun 的状态（Generated/Submitted/Approved/Rejected），若无任何 Run 则标记为"未报送"。

---

### User Story 2 - 一周内到期报表高亮提醒 (Priority: P1)

作为 Maker 用户，我希望看板对距截止日期不足 7 天的报表进行醒目的视觉提醒（如颜色标记、图标），以便我优先处理即将到期的报表。

**Why this priority**: 提醒功能是防止报表逾期的核心手段，与看板并列为最高优先级。

**Independent Test**: 可以通过配置一个截止日期在当前日期 7 天内的报表，确认其在看板上被高亮/标注为"即将到期"来独立测试。

**Acceptance Scenarios**:

1. **Given** 一个报表的截止日期在未来 7 天内（含）且该报表尚未完成审批流程，**When** 用户查看看板，**Then** 该报表卡片以橙色/黄色"即将到期"样式高亮显示，并标注剩余天数。
2. **Given** 一个报表的截止日期已经过去且未完成审批，**When** 用户查看看板，**Then** 该报表卡片以红色"已逾期"样式高亮显示，并标注逾期天数。
3. **Given** 一个报表的截止日期在 7 天以后，**When** 用户查看看板，**Then** 该报表卡片以正常样式显示，无特殊高亮。
4. **Given** 一个报表在截止日期前已通过审批（Approved），**When** 用户查看看板，**Then** 该报表卡片以绿色"已完成"样式显示，不再触发到期提醒。

---

### User Story 3 - 从看板快速跳转执行报表 (Priority: P2)

作为 Maker 用户，我希望能从看板中即将到期的报表卡片直接跳转到执行/提交该报表的操作页面，减少手动查找的步骤。

**Why this priority**: 提升操作效率，减少从"看到提醒"到"实际操作"的步骤数，是看板的延伸价值。

**Independent Test**: 可以通过点击看板上某个报表卡片的"去执行"按钮，确认跳转至对应报表的 Maker 操作视图来独立测试。

**Acceptance Scenarios**:

1. **Given** Maker 用户在看板上看到一个状态为"未报送"的报表卡片，**When** 点击卡片上的"去执行"按钮，**Then** 页面跳转至该报表的执行界面（Maker 视图并预选该报表）。
2. **Given** Maker 用户在看板上看到一个状态为"Generated"（已执行未提交）的报表卡片，**When** 点击卡片上的"去提交"按钮，**Then** 页面跳转至该报表 Run 的提交操作界面。

---

### User Story 4 - 报表截止日期管理 (Priority: P2)

作为管理员（Admin），我希望能为现有报表模板配置报送截止日期和报送频率，以便系统知道哪些报表需要在何时前完成报送。

**Why this priority**: 截止日期数据是看板和提醒的基础，但可以先通过数据库预置来启动 P1，因此配置管理为 P2。

**Independent Test**: 可以通过管理界面设置某报表的截止日期，然后在看板上确认其显示正确来独立测试。

**Acceptance Scenarios**:

1. **Given** 管理员打开报表截止日期管理界面，**When** 为一个报表设置截止日期（如 2026-04-30）和报送频率（月报），**Then** 系统保存配置并在看板上展示该报表及其截止日期。
2. **Given** 管理员修改了某报表的截止日期，**When** 看板刷新，**Then** 看板上该报表的截止日期和剩余天数随之更新。
3. **Given** 管理员为某报表设置了周期性报送（如每月），**When** 当期报送完成（Approved）后，**Then** 系统自动生成下一个周期的截止日期。

---

### Edge Cases

- 报表截止日期恰好为今天（剩余 0 天）时，应视为"即将到期"而非"已逾期"，提醒级别为最高。
- 报表在截止日期当天被提交但尚未审批时，看板状态应显示"已提交待审"而非"已逾期"。
- 同一报表存在多次 Run（如被拒绝后重新执行）时，看板只取最新一次 Run 的状态作为报送状态。
- 报表被标记为已删除（is_deleted = 1）时，不应出现在看板中。
- 多个报表在同一天到期时，看板应全部展示并按紧急程度排序。
- 截止日期未配置的报表不在看板中展示。
- 当管理员禁用某个报送计划（enabled = false）时，该报表立即从看板中消失。
- 截止日期判定统一使用服务器本地日期，不涉及时区转换。
- 当报表计划数量较大时，看板以分页形式展示（每页默认 20 条），暂不要求虚拟滚动。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供一个独立的"报送监控看板"页面，所有已登录用户（Maker/Checker/Admin）均可访问。
- **FR-002**: 系统必须支持为报表模板配置报送截止日期（deadline），包括单次截止日期和周期性截止日期（日报/周报/月报/季报/年报）。
- **FR-003**: 看板必须展示每个已配置截止日期的报表的名称、描述、截止日期、距截止日剩余天数、以及当前报送状态。
- **FR-004**: 系统必须对距截止日期 7 天以内（含）的未完成报表进行视觉高亮提醒（即将到期：橙色/黄色）。
- **FR-005**: 系统必须对已超过截止日期且未完成审批的报表标记为"已逾期"（红色高亮）。
- **FR-006**: 系统必须对已在截止日期前完成审批的报表标记为"已完成"（绿色），不再触发提醒。
- **FR-007**: 看板列表默认按紧急程度排序：已逾期 > 即将到期（按剩余天数升序） > 正常。
- **FR-008**: Maker 用户必须能从看板直接跳转至对应报表的执行/提交操作页面。
- **FR-009**: 当报表配置了周期性报送频率时，当期完成审批后系统必须自动滚动到下一个截止日期。
- **FR-010**: 看板必须过滤掉已删除的报表模板（is_deleted = 1）和未配置截止日期的报表。
- **FR-011**: 报送状态判定逻辑：基于当前截止周期内最新一次 ReportRun 的状态——无 Run 则"未报送"，Generated 则"已执行未提交"，Submitted 则"已提交待审"，Approved 则"已完成"，Rejected 则"被拒绝需重新执行"。

### Key Entities

- **ReportSchedule（报表报送计划）**：关联到报表模板，记录报送截止日期、报送频率（单次/日/周/月/季/年）、当前周期的截止日期、是否启用。一个报表模板最多关联一个有效的报送计划。
- **Report（报表模板，已存在）**：通过关联 ReportSchedule 获取截止日期信息。
- **ReportRun（报表运行实例，已存在）**：通过 reportId + 时间范围查找当前周期内最新的 Run 来确定报送状态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户登录后可在 3 次点击内到达报送监控看板并看到所有即将到期的报表。
- **SC-002**: 所有距截止日期 7 天以内的未完成报表在看板上 100% 被高亮标记。
- **SC-003**: 已逾期报表在看板上 100% 被红色标记且排列在最前面。
- **SC-004**: Maker 用户从看板点击到报表执行页面不超过 1 次额外点击。
- **SC-005**: 看板页面在 50 个报表计划的数据量下加载时间不超过 2 秒。
- **SC-006**: 周期性报送完成后，下一个周期截止日期在审批完成后 1 分钟内自动生成。

## Assumptions

- 复用现有的 JWT 认证体系，看板页面使用已有的登录态访问。
- 报送截止日期初始可通过数据库直接预置，管理界面为 P2 可后续迭代。
- 当前周期的判定范围：从上一个截止日期（或报送计划创建时间）到当前截止日期之间。
- 看板不需要实时推送通知（如 WebSocket），用户打开/刷新页面时获取最新数据即可。
- 看板对所有角色（Maker/Checker/Admin）展示相同的报表列表，但操作按钮根据角色不同而不同（Maker 可跳转执行，Checker 可跳转审批）。
- 报送频率类型限定为：ONCE（单次）、DAILY（日报）、WEEKLY（周报）、MONTHLY（月报）、QUARTERLY（季报）、YEARLY（年报）。
